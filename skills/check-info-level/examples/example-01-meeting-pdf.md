# 例 1: 教授会議事録 PDF の機密度判定

## 入力

```
/check-info-level /Users/staff/Documents/2026-04-15-faculty-meeting.pdf
```

## skill の処理

1. `Read` で PDF 本文を抽出
2. `Bash(file <path>)` で MIME 確認 → application/pdf
3. `Bash(pdfinfo <path>)` でメタデータ取得 → 作成者: 田中太郎、更新者: 山田花子
4. 本文評価:
   - 出席教員 15 名の発言記録（氏名あり）
   - 次年度カリキュラム改編（未確定）
   - 人事案件: 准教授候補 3 名の評議
5. 最高 Level を採用 → Level 3（人事案件含む）
6. ファイル固有警告: メタデータに作成者・更新者氏名残存

## 期待される出力

```markdown
## 判定結果: Level 3

**判定理由:**
本文に准教授候補 3 名の人事評議内容が含まれており、これは個人情報保護法上の要配慮個人情報には該当しないものの、流出時に当該候補者および大学の重大な不利益を生じる「機密情報」に該当する。出席教員 15 名の実名発言記録も Level 3 相当。

**AI 入力可否:**
原則不可。本文をそのまま AI に入力することは推奨しない。

**推奨次アクション:**
- [ ] 人事案件の議事部分（推定 pp.4-7）を完全に削除する
- [ ] 残部（カリキュラム改編議論）について、教員氏名を「A 教授」「B 准教授」等に置換する
- [ ] 置換済みテキストを Enterprise / API 環境で再判定 → Level 2 へ降格可能
- [ ] 上記が困難な場合、AI 利用を断念し人手で要約

**ファイル固有警告:**
- PDF メタデータに作成者「田中太郎」、最終更新者「山田花子」が残存。本文を匿名化しても、ファイル属性から作成者が特定される可能性がある。テキスト抽出してから入力を推奨
- 30 ページの PDF 全体に複数議題が混在している可能性。要約目的でも全文を渡すと、AI は人事議事も要約に含めてしまう

**根拠:** `skills/confidential-info-guidelines/SKILL.md` §3, §3.5
```

## なぜこの応答が適切か

- 判定理由を本文の具体記述（「准教授候補 3 名の人事評議」）に基づいて述べている
- 推奨アクションが具体的（pp.4-7 削除、A 教授 / B 准教授に置換）で実行可能
- §3.5 の PDF メタデータリスクを見落としていない
- 「Enterprise なら全 Level OK」と誤解を生む応答にしていない（人事案件は Enterprise でも避ける）
