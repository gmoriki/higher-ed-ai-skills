## 1. なぜ大学業務のAI活用は"翻訳"が必要なのか

### 1.1 大学の暗黙知がAIに渡らない構造的理由

大学業務の現場には、文書化されていない知識が膨大に存在する。

たとえば、稟議書の通し方ひとつとっても、「事前に関係部署の担当者に根回ししてから起案する」「学部長決裁と学長決裁で添付資料の粒度が違う」といった暗黙のルールがある。教授会の運営にしても、「この議題は報告事項として出すが、実質的には審議を求めている」「○○委員会の答申が先に出ていないと議論が進まない」といった空気の読み方が、業務の成否を左右する。学年暦の例外処理 -- 台風による休講判断、入試日程との調整、祝日授業実施日の設定 -- も同様に、マニュアルには書き切れない判断の積み重ねで成り立っている。

AIにこれらの業務を支援させたいと考えたとき、最初に直面する壁は「何をプロンプトに書けばよいかわからない」という問題ではない。「そもそも何が暗黙知なのか、当事者自身が意識していない」という構造的な問題である。長年の経験で身体化された判断基準は、言語化しなくても業務が回ってきたからこそ、言語化する動機がなかった。

本リポジトリの skill.md は、この「暗黙知の言語化」を一つの使命としている。AIが業務を直接遂行するのではなく、担当者がAIを使って業務を進める際に必要な「翻訳」-- 暗黙知をプロンプトや判断基準として明文化する作業 -- を支援するための道具である。

### 1.2 科学技術社会論（STS）の視点

科学技術社会論（STS: Science and Technology Studies）の知見が示すように、技術の導入は社会構造との相互作用のなかで成功も失敗も決まる。同じツールを導入しても、組織の文化・権限構造・意思決定プロセスによって、定着する場合としない場合がある。

大学という組織には、他のセクターにはない特殊性がある。

- **二重の権限構造**: 教員は学問の自由に基づく自律性を持ち、職員は事務組織のヒエラルキーのなかで動く。同じ「業務改善」でも、教員に対して標準化を強制することは難しい。
- **合議制の意思決定**: 教授会、各種委員会、部局長会議など、合議による意思決定が組織運営の基盤にある。トップダウンで「AIを使え」と号令をかけても浸透しにくい。
- **年度サイクルへの強い拘束**: 予算、人事、カリキュラムのすべてが年度単位で動く。年度途中での大幅な業務変更は、制度的にも心理的にもハードルが高い。

これらの特殊性を無視して「AIが使えるかどうか」だけを問うても意味がない。問うべきは「この業務文脈でAIが機能するか」である。本リポジトリのスキルは、この問いに一つずつ答えていく試みである。

## 2. skill.md 設計の5原則（森木版）

本リポジトリにおけるスキル設計は、以下の5つの原則に基づく。

### 原則1: Progressive Disclosure（段階的開示）

スキルの情報量は、利用者の必要度に応じて段階的に開示する。

| 段階 | 要素 | 役割 | ディレクトリ構成 |
|------|------|------|------------------|
| 第1段階 | `description`（冒頭の説明文） | トリガー：このスキルが何に使えるかを即座に伝える | SKILL.md 冒頭 |
| 第2段階 | 本体（判断ロジック・手順） | 実行：具体的に何をすべきかを示す | SKILL.md 本体 |
| 第3段階 | `references/` | 補足：根拠資料、比較表、外部リンク | `references/` ディレクトリ |

たとえば `skills/confidential-info-guidelines/` では、冒頭の description で「AIサービスに入力してよい情報の判断」という用途を示し、本体で判断フローを記述し、`references/` に各AIサービスのデータポリシー比較表を格納する。利用者は必要な深さまでだけ読めばよい。

この設計は、AIランタイムがスキルを選択する際にも機能する。ランタイムは第1段階の description だけを読んでスキルの適用可否を判断し、選択後に初めて本体を読み込む。不要な情報の読み込みを避けることで、コンテキストウィンドウの効率的な利用にもつながる。

### 原則2: description はトリガー設計

description は「このスキルの要約」ではなく「このスキルが起動されるべき状況の記述」として書く。

設計する際に想定するのは、利用者の発話パターンである。「機密情報をAIに入力していいか判断したい」「個人情報の取り扱いルールを確認したい」「AIサービスのデータポリシーを比較したい」-- こうした自然言語の問いかけに対して、AIランタイムが「このスキルを使うべきだ」と判断できるように書く。

description の書き方で避けるべきパターン:

- 抽象的すぎる記述（例: 「情報管理に関するスキル」）
- 技術用語の羅列（例: 「PII分類・DLP・コンプライアンスチェック」）
- 網羅的すぎる記述（例: あらゆるユースケースを列挙する）

逆に有効なパターン:

- 具体的な業務場面の記述（例: 「大学業務でAIサービスを利用する際に、入力してよい情報とそうでない情報を判断する」）
- 利用者が実際に使いそうな言い回しの組み込み

### 原則3: 本体は判断ロジック、詳細は references へ

SKILL.md の本体は500行以内を目安とする。本体には、実行可能な手順と判断基準だけを記述する。

背景説明、法令の条文、サービスごとの詳細比較、研究知見の引用などは `references/` ディレクトリに分離する。本体から references の該当ファイルへリンクを張ることで、必要な人だけが深掘りできるようにする。

この分離には実務上の理由もある。AIランタイムのコンテキストウィンドウには上限がある。本体を簡潔に保つことで、ランタイムが必要な情報を効率的に処理できる。詳細資料は、ランタイムが追加情報を必要としたときに references から取得する。

### 原則4: 模範回答を必ず含める

すべてのスキルには、`examples/` ディレクトリに模範回答を格納する。

模範回答は、AIが「このスキルで期待される出力のトーンと深さ」を理解するためのアンカーとなる。いわゆる few-shot prompting の原理と同じで、期待される出力の具体例を示すことで、AIの応答品質を安定させる。

模範回答に含めるべき要素:

- 利用者の具体的な問いかけ（入力例）
- スキルを適用した場合の理想的な応答（出力例）
- なぜその応答が適切かの簡潔な解説

模範回答が古くなった場合は、本体の改訂時にあわせて更新する。

### 原則5: 陳腐化前提の更新設計

AIサービスの仕様変更、法改正、大学を取り巻く環境の変化により、スキルの内容は必ず陳腐化する。これを前提として、すべてのスキルに以下のメタデータを付与する。

- `version`: セマンティックバージョニング（例: v1.0.0）
- `last_updated`: 最終更新日

更新ポリシーの詳細は第5章および `docs/update-policy.md` で述べる。

## 3. 大学業務固有の注意点

skill.md を大学業務に適用する際には、他のセクターにはない制約と配慮が必要になる。以下に主要な4つの注意点を整理する。

### 3.1 単年度予算制

日本の国公私立大学は、原則として単年度予算制で運営されている。これはAI導入に対して以下の制約を課す。

- **導入タイミングの制約**: 新しいツールやサービスの契約は、原則として年度初めの予算に計上する必要がある。年度途中で「このAIサービスを使いたい」と思っても、予算措置が間に合わない場合がある。
- **継続利用の不確実性**: 単年度予算では、翌年度の予算が確保される保証がない。AIサービスの年間契約を前提としたスキルを作っても、翌年度に予算がつかなければ使えなくなる。
- **費用対効果の説明責任**: 予算要求時に「AIを使うことでどれだけ業務が効率化されるか」を定量的に示す必要がある。

スキル設計においては、無料で利用できるサービスを前提とするか、有料サービスを前提とする場合はその旨を明記し、代替手段も併記する。

### 3.2 教員と職員の権限分離

大学組織には、教員と職員という二つの系統がある。同じ業務領域であっても、意思決定の権限は両者で大きく異なる。

- **教務関連**: カリキュラム編成は教員の権限（教授会の審議事項）。時間割作成やシラバス入力の実務は職員が担う。
- **研究支援**: 研究費の使途決定は教員（研究代表者）。経理処理や報告書のとりまとめは職員が行う。
- **学生対応**: 成績評価は教員の専権事項。履修相談や奨学金手続きは職員が対応する。

スキルを設計する際には、「このスキルは誰が使うのか」を明確にする必要がある。教員向けのスキルと職員向けのスキルでは、前提となる権限、アクセスできる情報、期待される出力形式が異なる。`domain-skills/` ディレクトリでは業務領域ごとにスキルを整理しているが、各スキルの冒頭で想定利用者を明示する方針をとる。

### 3.3 学生データの取り扱い

大学が保有する学生データ -- 成績、出席状況、履修情報、奨学金受給状況、健康情報、相談記録など -- は、最も慎重な取り扱いが求められる情報群である。

スキル設計における原則:

- **AIサービスへの入力禁止を原則とする**: 学生の個人情報を外部AIサービスに入力することは、原則として行わない。これは各大学の個人情報保護規程およびAI利用ガイドラインに基づく。
- **匿名化・加工後の利用は別途判断**: 統計処理済みデータや、個人が特定できないよう加工したデータの利用は、各大学のルールに従って判断する。
- **学内閉域AIの活用**: Azure OpenAI Service や AWS Bedrock など、データが外部に送信されない環境を利用できる場合は、そのサービスのデータポリシーを確認したうえで判断する。

`skills/confidential-info-guidelines/` は、この判断を支援するためのスキルとして最初にリリースした。

### 3.4 個人情報保護法・各大学規程との整合性

個人情報保護法（2022年改正施行）への準拠は大前提であるが、大学ごとに独自の規程が存在する点に注意が必要である。

- **国立大学法人**: 独立行政法人等個人情報保護法（現在は個人情報保護法に統合）に基づく規程を整備している。
- **公立大学**: 設置自治体の個人情報保護条例に準拠する場合がある。
- **私立大学**: 各大学が独自の個人情報保護規程を定めている。

スキルでは「個人情報保護法を遵守すること」と記述するだけでは不十分で、「所属大学の個人情報保護規程およびAI利用ガイドラインを確認すること」という具体的なアクションを指示する必要がある。法令と大学規程の関係は複雑であるため、スキルの本体では判断の枠組みを示し、具体的な法令解釈は `references/` に委ねる。

## 4. ランタイム非依存の思想

### 4.1 skill.md を Single Source of Truth に

本リポジトリの skill.md は、Anthropic Claude の Custom Instructions / Skills の形式を基盤として記述する。しかし、スキルの内容自体は特定のAIランタイムに依存しない。

Claude の形式を採用する理由は、description / 本体 / references という3層構造が、第2章で述べた Progressive Disclosure の原則と自然に合致するためである。形式としての採用であり、Claude でしか使えないスキルを作ることが目的ではない。

skill.md が Single Source of Truth（唯一の正）であるとは、以下を意味する:

- スキルの内容を更新する際は、まず skill.md を更新する
- 他のランタイム向けの変換版は、skill.md から派生させる
- skill.md と変換版の内容に矛盾がある場合、skill.md が正である

### 4.2 他ランタイムへの変換前提

現実として、日本の大学職員の多くは ChatGPT または Microsoft Copilot を利用している。Claude を使っている職員は少数派である。「Claude でしか使えないスキル集」では、公開リポジトリとしての存在意義が限定される。

そこで、`runtime-adapters/` ディレクトリに、skill.md を他のランタイムで活用するための変換ガイドを提供する。

想定する変換先:

| ランタイム | 変換方式 | 備考 |
|-----------|---------|------|
| ChatGPT（GPTs） | Custom Instructions + Knowledge files に変換 | description -> Instructions、references -> Knowledge |
| Microsoft Copilot | Copilot Studio のトピック構成に変換 | 大学のMicrosoft 365環境での利用を想定 |
| GitHub Copilot | Agent mode の指示文として変換 | 開発業務を行う職員向け |
| ローカルLLM | system prompt として変換 | データを外部に出せない業務向け |

変換ガイドは skill.md の構造と各ランタイムの仕様の対応関係を示すものであり、完全な自動変換を保証するものではない。スキルの内容によっては、変換先のランタイムの制約に合わせて手動での調整が必要になる場合がある。

### 4.3 ランタイム非依存であることの限界

ランタイム非依存を志向するとはいえ、完全な中立性は現実的ではない。各ランタイムによって、コンテキストウィンドウの長さ、ツール呼び出しの有無、マルチモーダル対応の範囲などが異なる。これらの差異はスキルの実効性に影響する。

本リポジトリでは、この限界を認めたうえで、「内容の記述はランタイム非依存に保ち、実行環境への適応は runtime-adapters/ に委ねる」という分離方針を採る。

## 5. 更新と陳腐化との戦い方

### 5.1 半期改訂ポリシー

すべてのスキルは、原則として半期（6か月）ごとに内容を確認し、必要に応じて改訂する。

- **3月末改訂**: 年度末。新年度に向けた法改正、サービス仕様変更、組織改編への対応。
- **9月末改訂**: 年度中間。上半期の運用実績を踏まえた修正。後期授業開始に合わせた調整。

改訂の結果、内容に変更がない場合でも、`last_updated` を更新し「確認済み」であることを明示する。利用者が「この情報はいつ時点のものか」を常に判断できるようにするためである。

### 5.2 AI進化速度と鮮度保証

半期改訂だけでは対応できない変化がある。以下の事象が発生した場合は、臨時改訂を行う。

- **AIサービスのデータポリシー変更**: 主要サービス（ChatGPT、Copilot、Claude、Gemini）のデータ取り扱い方針が変更された場合。特に、学習データへの利用に関する方針変更は `skills/confidential-info-guidelines/` に直接影響する。
- **法改正**: 個人情報保護法、著作権法、AI関連の新規立法や改正が施行された場合。
- **文部科学省通知・ガイドライン**: 大学におけるAI利用に関する通知やガイドラインが発出された場合。
- **重大インシデント**: 大学業務におけるAI利用で重大な事故や問題が発生し、スキルの記述内容に影響がある場合。

臨時改訂の場合も、通常の改訂と同じく CHANGELOG.md に記録する。

### 5.3 CHANGELOG.md の運用

すべての変更は `CHANGELOG.md` に記録する。形式は [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) に準拠する。

CHANGELOG.md が果たす役割:

- **利用者**: 前回確認時からの差分を把握できる。自分が使っているスキルに変更があったかを確認できる。
- **貢献者**: プロジェクトの変更履歴を追え、どの方向に進化しているかを理解できる。
- **引用者**: 特定バージョンのスキルを引用している場合、そのバージョン以降の変更を確認できる。

### 5.4 deprecated/ への移動

陳腐化したスキルは削除せずに `deprecated/` ディレクトリへ移動する。

削除ではなく移動とする理由:

- **過去の参照者保護**: すでにスキルを参照・引用している文書やシステムが存在する可能性がある。削除するとリンク切れが発生する。
- **履歴としての価値**: 「かつてこういう判断基準があった」という記録自体に価値がある。特に、法改正やサービス仕様変更によって内容が変わった場合、変更前の記録は参考になる。
- **復活の可能性**: 一度陳腐化したスキルでも、状況の変化により再び有効になる場合がある。

`deprecated/` に移動する際は、以下を行う:

1. スキルの冒頭に非推奨である旨と、代替スキル（ある場合）へのリンクを記載する
2. CHANGELOG.md に移動の記録を残す
3. 元のパスからは移動先へのリダイレクト説明を残す（README等で案内）
